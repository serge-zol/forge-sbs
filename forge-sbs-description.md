## 1. Суть проекту

**Кузня** — це фабрика AI-агентів. Тут створюються, рецензуються, версіонуються і зберігаються інструкції (system prompts) для AI-агентів. Ціль — надійна, перевірена бібліотека інструкцій, яку можна розгортати на будь-якій платформі: Claude, GPT, Gemini, Anthropic API та ін.

Проект будується крок за кроком: від першого мета-агента (Коваль) до повноцінної системи спеціалізованих агентів, кожен з яких вирішує конкретну задачу.

## 3. Агенти — ядро системи

### Коваль (Forge Master)
**Роль:** Мета-агент. Пише інструкції для всіх інших агентів.
**Статус:** inbox (очікує обробки)
**Специфіка:** Найважливіший агент системи — від якості його інструкцій залежить якість всіх інших. Перша версія генерується через Opus 4.8 незалежно від наявних чернеток.

### Ментор (Mentor)
**Роль:** Навчає процесу створення інструкцій. Орієнтований на людину — пояснює принципи, методологію, типові помилки.
**Статус:** inbox (очікує обробки)
**Специфіка:** Не генерує інструкції сам — навчає як їх будувати. Може використовуватись для онбордингу або самоперевірки.

### ЧЕК (Checker)
**Роль:** Незалежний рецензент. Перевіряє якість інструкцій перед деплоєм.
**Статус:** inbox (очікує обробки)
**Специфіка:** Працює НЕЗАЛЕЖНО від Коваля — оцінює за критеріями якості, шукає прогалини, протиріччя, амбігуїтет. Не знає хто писав.

### Спеціалізовані агенти (майбутні)
Будуть додаватись згідно поточних задач і ТЗ. Кожен — окрема папка в `agents/`, проходить повний цикл GENERATE → REVIEW → TEST → STORE.

## 4. Воркфлоу

```
INPUT → GENERATE → REVIEW → TEST → STORE
```

| Крок | Хто | Що відбувається |
|------|-----|-----------------|
| **INPUT** | Серж (Dispatch/телефон) | Описує агента: що робить · для кого · контекст. 2-3 речення. |
| **GENERATE** | Наві → Opus 4.8 | Code task → Opus 4.8 + extended thinking → XML-файл у `inbox/` |
| **REVIEW** | ЧЕК | Незалежна перевірка. Не знає хто писав. Шукає дефекти. |
| **TEST** | Наві | Запуск `<test_cases>` із файлу. Виявлення де агент "ламається". |
| **STORE** | Наві | `agents/[name]/vN/` + git commit + оновлення `registry.md` |

### Захист від системних ризиків

| Ризик | Захід |
|-------|-------|
| Кругова залежність (ЧЕК не верифіковано ззовні) | Коваль і ЧЕК перевіряються незалежно одне від одного. Перша версія кожного — Opus 4.8 без опори на наявні чернетки |
| Розсинхронізація full/core | `full.md` і `core.md` завжди в одному git commit |
| Деградація без версіювання | Кожна зміна = новий `vN/` + запис у `meta.yaml` |
| Суб'єктивна оцінка без тестів | Кожен агент має `<test_cases>`: мін. 3 сценарії (нормальний / поганий вхід / крайній кейс) |

## 5. Технічний стек

### Ведуча модель

| Параметр | Значення | Обґрунтування |
|----------|----------|---------------|
| model | `claude-opus-4-8` | Найвища якість для мета-задач. Невеликий приріст якості тут → великий ефект на всіх агентах |
| extended_thinking | `true` | Модель аналізує конфлікти в вимогах перед формулюванням. Без цього — текст, не специфікація |
| budget_tokens | `10000` | Достатньо для глибокого аналізу складних агентів |
| temperature | `0.3` | Точність і відтворюваність. Не 0 (надто жорсткий), не вище 0.5 |
| max_tokens | `8000` | Повні інструкції + test cases |
| output_format | XML | Стабільний машинний вивід: `<system_prompt>`, `<core>`, `<notes>`, `<test_cases>` |

### Виклик: Python + Anthropic SDK → Claude Code task

### Версіонування: GitHub (`serge-zol/forge-sbs`)

⚠️ `claude-fable-5` — недоступна з 12.06.2026 (урядова директива США). Не використовувати.

## 6. Документація (OpenSpec)

Кожен агент матиме `spec.md` — структурований опис за форматом:

```markdown
# [Назва агента] · Специфікація

## Призначення
## Вхідні дані (Input)
## Вихідні дані (Output)
## Обмеження
## Тест-кейси
## Версії та зміни
## Відомі обмеження
```

Цей формат дозволяє:
- Швидко зрозуміти агента без читання повних інструкцій
- Порівнювати версії
- Інтегрувати в інші системи
- Передавати іншим людям/інструментам

## 7. Управління

**Серж** керує через Dispatch з телефона (переважно 19:00–23:00):
- описує задачу
- затверджує або відхиляє результати
- дає дозвіл на незворотні дії (git push, перезапис, видалення)

**Наві (Claude)** виконує всю технічну роботу:
- генерація інструкцій
- файлові операції
- git операції (локальний commit)
- оновлення реєстру і досвіду

**git push на GitHub** — тільки після явного дозволу Сержа.

## 8. Правила Наві (обов'язкові)

1. На початку кожної сесії читати `experience.md` і `registry.md`
2. Після кожної значущої операції оновлювати `experience.md`
3. Після зміни статусу агента — одразу оновлювати `registry.md`
4. `full.md` і `core.md` — завжди синхронно
5. Без явного дозволу — не виходити за межі `forge-sbs/`
6. Незворотні дії (delete, push, перезапис) — спочатку план, потім підтвердження
