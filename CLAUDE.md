# Кузня. Крок за кроком. · forge-sbs

## Суть проекту
Кузня — фабрика AI-агентів. Тут створюються, перевіряються, версіонуються і зберігаються інструкції (system prompts) для агентів. Ціль — надійна бібліотека перевірених інструкцій, розгортуваних на будь-якій платформі (Claude, GPT, API).

Серж керує через Dispatch з телефона (19:00–23:00). Наві (Claude) виконує всю технічну роботу.

## Ведуча модель

| Параметр | Значення | Примітка |
|----------|----------|----------|
| model | `claude-opus-4-8` | |
| thinking | `{"type": "enabled"}` | extended thinking |
| effort | `"high"` | замінює `budget_tokens` для Opus 4.8 |
| temperature | ❌ не вказувати | несумісно з thinking; фіксується на 1 |
| max_tokens | `32000` | має перевищувати thinking budget; до 128k |
| output_format | XML | |

Виклик: Python + Anthropic SDK → code task в Claude Code.
⚠️ `claude-fable-5` — недоступна (урядова директива США 12.06.2026, підтверджено anthropic.com). Не використовувати.

## Воркфлоу

```
INPUT → GENERATE → REVIEW → TEST → STORE
                       ↑         |
                       └─ FAIL ──┘
```

1. **INPUT** — Серж описує агента в Dispatch: що робить · для кого · контекст
2. **GENERATE** — Наві: code task → Opus 4.8 + extended thinking → XML → `inbox/` або `agents/[name]/vN/`
3. **REVIEW** — ЧЕК (інша модель: GPT-4o / Gemini) перевіряє незалежно; FAIL → назад у GENERATE з фідбеком
4. **TEST** — LLM-суддя (Opus 4.8) оцінює за критеріями: precision, completeness, no-ambiguity; FAIL → GENERATE з фідбеком
5. **STORE** — `agents/[name]/vN/` + git commit + оновити `registry.md`

### Захист від слабких місць

| Ризик | Захід |
|-------|-------|
| Bootstrap (хто пише v1 Коваля/ЧЕКа) | Seed-версія від людини або базової моделі; далі — самопокращення через Коваля |
| Кругова залежність / уявна незалежність ЧЕКа | ЧЕК = інша модель (GPT-4o або Gemini) — різні сліпі зони, реальна незалежність |
| Розсинхронізація full/core | `full.md` і `core.md` оновлюються в одному коміті. Заборонено оновлювати один без іншого |
| Відсутність версіювання | Кожна зміна = новий `vN/` + запис у `meta.yaml` |
| Відсутність тест-кейсів | Кожен агент має `<test_cases>` з ≥3 сценаріями (хороший / поганий / крайній кейс) |

## Правила для Наві (обов'язкові)

### `experience.md` — Досвід проекту
- **ЧИТАТИ** на початку кожної сесії в цьому проекті
- **ОНОВЛЮВАТИ** після кожної значущої операції: що спрацювало / що ні / що виявили
- **ЗАСТОСОВУВАТИ**: є прецедент — використовуй, не винаходь знов
- Формат запису: `[YYYY-MM-DD] Тема: ...`

### `registry.md` — Реєстр агентів
- **ОНОВЛЮВАТИ** одразу після зміни статусу або появи нової версії
- Поля: назва · версія · статус · дата · опис
