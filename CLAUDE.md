# Кузня. Крок за кроком. · forge-sbs

## Суть проекту
Кузня — фабрика AI-агентів. Тут створюються, перевіряються, версіонуються і зберігаються інструкції (system prompts) для агентів. Ціль — надійна бібліотека перевірених інструкцій, розгортуваних на будь-якій платформі (Claude, GPT, API).

Серж керує через Dispatch з телефона (19:00–23:00). Наві (Claude) виконує всю технічну роботу.

| Параметр | Значення |

## Воркфлоу

```
INPUT → GENERATE → REVIEW → TEST → STORE
```

1. **INPUT** — Серж описує агента в Dispatch: що робить · для кого · контекст
2. **GENERATE** — Наві: code task → Opus 4.8 + extended thinking → XML → `inbox/` або `agents/[name]/vN/`
3. **REVIEW** — ЧЕК перевіряє незалежно від генератора
4. **TEST** — запуск `<test_cases>` із файлу; виявлення прогалин
5. **STORE** — `agents/[name]/vN/` + git commit + оновити `registry.md`

### Захист від слабких місць

| Ризик | Захід |
|-------|-------|
| Кругова залежність | Коваль і ЧЕК перевіряються незалежно; перша версія — Opus 4.8 без опори на попередні інструкції |
| Розсинхронізація full/core | `full.md` і `core.md` оновлюються в одному коміті |
| Відсутність версіювання | Кожна зміна = новий `vN/` + запис у `meta.yaml` |
| Відсутність тест-кейсів | Кожен агент має `<test_cases>` з ≥3 сценаріями |

## Правила для Наві (обов'язкові)

### `experience.md` — Досвід проекту
- **ЧИТАТИ** на початку кожної сесії в цьому проекті
- **ОНОВЛЮВАТИ** після кожної значущої операції: що спрацювало / що ні / що виявили
- **ЗАСТОСОВУВАТИ**: є прецедент — використовуй, не винаходь знов
- Формат запису: `[YYYY-MM-DD] Тема — висновок`

### `registry.md` — Реєстр агентів
- **ОНОВЛЮВАТИ** одразу після зміни статусу або появи нової версії
- Поля: назва · версія · статус · дата · опис

### Загальне
- Без явного дозволу — не виходити за межі `sb-projects/forge-sbs/`
- Незворотні дії (delete, git push, перезапис) — план + підтвердження Сержа
- Якщо `experience.md` суперечить поточному запиту — зупинитись і уточнити
